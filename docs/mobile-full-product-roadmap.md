# Nane Native Mobile Full Product Roadmap

Created: 2026-07-04

## Purpose

This document is the execution roadmap for turning the Nane mobile app into a complete native iOS and Android experience.

The mobile app must not be a web wrapper. Every migrated page should become a native mobile workflow that uses the shared mobile design system, real backend APIs, role permissions, and proper mobile interaction patterns.

## Current State

Completed foundation:

- Shared native mobile design system.
- Shared API client and auth/session foundation.
- Role-aware route registry for Association Admin, Member Portal, and System Admin.
- Native association dashboard foundation.
- Native member dashboard foundation.
- Native members list.
- Native member detail screen.
- iOS simulator verification for dashboard to members to member detail.

Current committed baseline:

- Commit: `a60ac77 Build native mobile foundation and members flow`

## Global Migration Rule

Migrate one route or tightly connected route group at a time.

Do not start the next page until the current page has:

- Native screen implementation.
- Real API integration.
- Shared mobile components only.
- No fake data.
- Loading, empty, error, and success states.
- Permission handling.
- Light mode verification.
- Dark mode verification when colors/status/surfaces are involved.
- iOS simulator screenshot captured after implementation.
- UI/UX review from the screenshot before completion.
- Any obvious visual, spacing, contrast, alignment, navigation, or flow issue fixed before completion.
- Typecheck and lint passing.

Android verification starts after the iOS foundation is stable.

## Definition Of Done For Every Mobile Page

Every page must satisfy this checklist:

- Uses `MobileScreen`.
- Uses `MobilePageHeader`.
- Uses `MobileKpiGrid` and `MobileKpiCard` for metrics.
- Uses `MobileDataList` instead of desktop tables.
- Uses `MobileStatusBadge` for statuses.
- Uses `MobileSearchToolbar` for list search.
- Uses `MobileStatusTabs` where status filtering exists.
- Uses `MobileFilterSheet` and `MobileSortSheet` for filters and sorting.
- Uses `MobileFormSection`, `MobileTextInput`, `MobileSelect`, `MobileDatePicker`, and `MobileAmountInput` for forms.
- Uses `MobileConfirmSheet` for destructive or irreversible actions.
- Uses `MobilePageLoadingState`, `MobileDetailSkeleton`, `MobileSkeletonList`, or `MobileFormSkeleton` for loading.
- Uses `MobileEmptyState` and `MobileErrorState`.
- Handles long names, long TZS amounts, long statuses, and small screens.
- Preserves existing backend behavior and permissions.
- Does not remove web-only features without documenting why.
- Has at least one simulator screenshot captured after the page is completed.
- Has screenshot-based UI/UX review completed before the route checkbox is marked done.
